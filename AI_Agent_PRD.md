# 工厂订单管理系统 — AI 智能体功能 PRD

**版本：** v1.0 　**状态：** 草稿 　**日期：** 2026-05-20

---

## 1. 项目背景

工厂老板原本依靠员工将客户订单手动录入 Excel，并由员工根据工厂机器情况人工安排生产排单。该流程重复性高、耗时耗力，且依赖人工经验判断，存在以下核心问题：

- 录入效率低：从客户发来消息到订单录入完成平均耗时 15–30 分钟
- 排单决策慢：需要人工综合考虑机器限制、订单规格、合并生产可能性，要想很久
- 人力成本高：重复性操作占用大量人力，无法专注于更高价值的业务判断

目前已开发完成一套基础订单管理与排单系统（Next.js + Prisma + MySQL），支持手动录入订单、排单看板管理等功能。本 PRD 定义在此基础上新增 AI 智能体模块的功能需求。

## 2. 目标与成功指标

### 2.1 产品目标

- 将订单录入时间从 15–30 分钟压缩至 3 分钟以内
- 将排单决策从纯人工判断升级为 AI 辅助决策，减少遗漏和错误
- 提升整体操作体验，从「操作系统填表」变成「和助手自然对话」

### 2.2 成功指标（KPI）

| 指标 | 当前基线 | 目标值 | 测量方式 |
|------|---------|--------|---------|
| 订单录入时长 | 15–30 分钟/张 | ≤ 3 分钟/张 | 系统日志 |
| AI 信息提取准确率 | — | ≥ 90% | AgentLog 统计 |
| 排单方案老板接受率 | — | ≥ 80% | 确认/调整比例 |
| 老板操作确认率 | — | ≥ 95% | confirm/cancel 比例 |
| AI 平均响应时间 | — | ≤ 5 秒 | API 响应时间 |

## 4. 功能范围

### 4.1 功能概览

AI 智能体以对话框形式嵌入现有系统（页面右下角悬浮按钮），通过单一对话入口处理所有任务，由 AI 自动识别意图并调用对应 Tool 执行。

核心功能分为两大模块：
- **模块 A：智能接单** — 对话式录入订单，替代手动填表
- **模块 B：智能排单** — AI 生成排单方案，对话式调整确认

### 4.2 不在本期范围内

- 微信公众号接入（后续版本考虑）
- 发货、回款、财务相关功能
- 多用户权限管理
- 语音转文字（使用浏览器原生语音输入）

## 5. 功能详细设计

### 5.1 对话框交互设计

#### 5.1.1 入口与布局

- 页面右下角固定悬浮绿色圆形按钮（AI 图标）
- 点击弹出对话框，宽 360px，高 520px，手机端全屏
- 顶部显示标题「AI 助手」+ 关闭按钮
- 中间区域：消息气泡列表（用户消息靠右，AI 消息靠左）
- 底部：文字输入框 + 发送按钮，支持换行
- 空状态展示快捷提示语：「录入新订单」、「帮我排单」、「查看待排单」

#### 5.1.2 任务切换机制

当老板在录单或排单进行中途发送其他意图的消息时，AI 应识别到任务切换，并主动提示：

> 「你当前有一个录单任务未完成，是继续录单，还是先处理排单？」

等待老板明确回复后再执行对应操作，不强制切换，不并行处理。

### 5.2 模块 A：智能接单

#### 5.2.1 完整交互流程

1. 老板复制微信消息粘贴到对话框，或直接语音/手打描述订单
2. AI 调用 `extract_order_info` 提取结构化字段
3. AI 调用 `query_customer` 模糊匹配客户，找到相似客户则直接使用，找不到则提示新建
4. AI 调用 `query_product` 匹配产品
5. 针对缺失字段逐一追问，话术自然，不像填表格，每次最多问一个问题
6. 配方选择：AI 提示三个选项（见 5.2.2）
7. 所有信息收集完毕后，展示订单摘要卡片供老板确认
8. 老板点击「确认创建」后，后台调用 API 创建订单，状态设为「待排单」

#### 5.2.2 配方选择逻辑

AI 在收集完产品和规格信息后，展示配方选择提示：

- **选项 1：暂不设置配方**（后续在订单详情里补充）
- **选项 2：从配方库选择**（AI 调用 `query_formula` 列出该产品的匹配配方，让老板选）
- **选项 3：新建配方**（AI 引导老板填写配方名称和原材料比例）

#### 5.2.3 客户匹配规则

| 情况 | 处理方式 |
|------|---------|
| 完全匹配 | 直接使用，无需提示 |
| 模糊匹配（相似度高） | 直接使用最相似的，在摘要卡片中展示供老板确认 |
| 找不到匹配 | 提示「系统中未找到该客户，是否新建？」 |
| 多个相似结果 | 列出候选让老板选择 |

#### 5.2.4 必填字段定义

| 字段 | 是否必填 | 说明 |
|------|---------|------|
| 客户 | 必填 | 模糊匹配或新建 |
| 产品 | 必填 | 从产品库匹配 |
| 规格参数（厚度/宽度等） | 必填 | 按产品类型决定必填项 |
| 数量 + 单位 | 必填 | kg 或 t |
| 配方 | 选填 | 三选一：不设置/选已有/新建 |
| 额外要求/备注 | 选填 | 自由文本 |

---

### 5.3 模块 B：智能排单

#### 5.3.1 完整交互流程

1. 老板发起排单请求（如「帮我排一下今天的单」）
2. AI 调用 `get_pending_orders` 展示所有待排单订单列表，让老板选择本次要排哪些
3. 老板确认订单范围后，AI 调用 `get_machine_status` 获取所有机器当前状态
4. AI 调用 `generate_schedule_plan` 生成排单方案
5. AI 用自然语言解释方案逻辑（为什么这样排、哪些单可以合并、理由是什么）
6. 老板对话式反馈调整意见，AI 调用 `adjust_schedule_plan` 修改方案
7. 步骤 6 可反复进行，直到老板满意
8. 老板确认最终方案后，AI 调用 `confirm_and_execute` 后台创建生产任务

#### 5.3.2 排单方案生成规则

- **机器匹配：** 按订单产品类型筛选可用机器，检查宽度是否在机器允许范围内
- **合并生产判断：** 同台机器、材质相同、宽度之和 ≤ 机器最大宽度时建议合并
- **排序原则：** 默认按接单时间先后排队，特殊情况老板可通过对话调整优先级
- **方案解释：** AI 必须用自然语言解释每一个排单决策的原因

#### 5.3.3 对话调整示例

> 老板：「把华兴那张单放到 3 号机」
>
> AI：「好的，我已将 ORD-20260520-003（华兴包装·PE膜·500kg）调整到 3 号机，排在当前队列第 2 位。更新后的方案如下：...」

> 老板：「那两张 PE 膜能合并吗」
>
> AI：「ORD-003（宽 300mm）和 ORD-005（宽 400mm）在 3 号机上宽度之和 700mm，未超过 3 号机最大宽度 1200mm，材质相同，可以合并生产。已为你更新方案。」

## 6. Tool 定义（Skills）

以下是 AI Agent 可调用的全部 Tool，每个 Tool 的 `description` 直接决定 AI 的意图识别准确性，需精心设计。

### 6.1 录单相关 Tools

| Tool 名称 | 触发场景 | 输入 | 输出 |
|-----------|---------|------|------|
| `extract_order_info` | 用户粘贴文字或描述订单信息时 | 原始文字 | 结构化字段 JSON |
| `query_customer` | 需要匹配客户时 | 客户名关键词 | 匹配结果列表 |
| `query_product` | 需要匹配产品时 | 产品名关键词 | 匹配产品列表 |
| `query_formula` | 老板选择从配方库选择时 | productId | 匹配配方列表 |
| `confirm_and_create_order` | 所有字段收集完毕，老板确认后 | 完整订单数据 | 创建结果 |
| `update_order` | 老板要修改已有订单时 | orderId + 变更字段 | 更新结果 |

### 6.2 排单相关 Tools

| Tool 名称 | 触发场景 | 输入 | 输出 |
|-----------|---------|------|------|
| `get_pending_orders` | 发起排单时获取待排单列表 | 无 | 待排单订单列表 |
| `get_machine_status` | 生成方案前获取机器状态 | 无 | 机器列表 + 当前队列 |
| `generate_schedule_plan` | 老板确认排单范围后生成方案 | 选中订单列表 | 排单方案 JSON |
| `adjust_schedule_plan` | 老板对话式调整方案时 | 当前方案 + 调整指令 | 更新后方案 |
| `confirm_and_execute` | 老板最终确认方案后执行 | 最终方案 | 创建生产任务结果 |

### 6.3 通用 Tools

| Tool 名称 | 触发场景 | 输入 | 输出 |
|-----------|---------|------|------|
| `check_unfinished_task` | 检测到意图切换时 | 无 | 未完成任务状态 |

## 7. System Prompt 设计原则

### 7.1 核心指令

- 你是一个塑料薄膜工厂的订单管理助手，服务对象是工厂老板
- 老板可能用口语、缩写表达（如「PE膜」「50厚」「600宽」），需正确理解
- 执行任何写操作前，必须先展示摘要卡片，等待老板明确确认后才执行
- 信息不完整时，逐一追问，每次最多问一个问题，不要瞎猜
- 查询类操作（读操作）可以直接执行，无需确认
- 排单方案必须解释决策逻辑，让老板理解为什么这样排
- 检测到任务切换时，主动提示并等待老板决策
- 所有回复用简洁、友好的中文

### 7.2 不应该做的事

- 不要在老板未确认前执行任何创建、修改、删除操作
- 不要强行猜测缺失信息，宁可追问
- 不要一次性问超过一个问题
- 不要生成过长的回复，保持简洁

## 8. 数据库新增表设计

### ChatMessage 表

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|---------|------|
| id | String (cuid) | 是 | 主键 |
| role | String | 是 | user / assistant / tool |
| content | Text | 否 | 消息文字内容 |
| toolCalls | Text (JSON) | 否 | AI 要调用的 Tool 列表 |
| toolCallId | String | 否 | Tool 调用 ID（结果消息用） |
| toolName | String | 否 | 被调用的 Tool 名称 |
| isPending | Boolean | 是 | 是否等待老板确认（默认 false） |
| createdAt | DateTime | 是 | 创建时间 |

### AgentLog 表（后续迭代加入）

- 记录每次 Tool 调用的输入参数、执行结果、是否成功
- 用于问题排查和后续 Prompt 优化的数据支持
- 字段：id、toolName、input（JSON）、output（JSON）、status、durationMs、createdAt

## 9. 评测标准（上线前必须通过）

### 9.1 测试用例集（Golden Set）

| # | 输入 | 期望行为 | 判断标准 |
|---|------|---------|---------|
| 1 | 「华兴包装要500kg PE膜」 | 提取客户+产品，追问规格 | 客户匹配正确，追问厚度/宽度 |
| 2 | 「帮我加个新客户，华东纸业，李总」 | 展示确认卡片，待老板确认 | 确认后才创建，不提前执行 |
| 3 | 「上次华兴那个规格，这次要800kg」 | 识别历史客户，追问本次规格变化 | 客户匹配正确，不遗漏数量 |
| 4 | 录单中途说「帮我排单」 | 识别任务切换，主动询问是否切换 | 不强制切换，等老板决定 |
| 5 | 「帮我排一下今天的单」 | 展示待排单列表，让老板选 | 列表完整，等老板确认范围 |
| 6 | 「把华兴那张放到3号机」 | 识别调整指令，更新方案 | 方案正确更新，说明原因 |
| 7 | 「那两张 PE 膜能合并吗」 | 检查合并条件，给出判断 | 正确计算宽度，给出可/否结论 |
| 8 | 「好，就这样排」 | 展示最终方案确认卡片 | 执行前再次确认，成功后提示完成 |

### 9.2 核心评测指标

- **信息提取准确率：** Golden Set 中字段提取正确率 ≥ 90%
- **意图识别准确率：** 正确调用 Tool 的比例 ≥ 90%
- **不误操作率：** 在未确认的情况下执行写操作次数 = 0（硬性要求）
- **任务完成率：** 测试用例从发起到成功创建订单/执行排单的完成率 ≥ 85%

## 11. 后续迭代方向

- **微信公众号接入：** 申请认证服务号后，将对话入口扩展到微信
